📋 项目概览

本项目是 Anthropic 官方发布的 Financial Services 插件套件，面向投资银行、股票研究、资产管理、私募股权、基金运营等金融机构的全场景 Claude Code 插件和 Managed Agent 模板集合。

项目以"同一源文件，两种交付方式"为核心理念：每个 Named Agent 既可作为 Claude Code 插件在交互式会话中即时调用，也可作为托管代理通过 Anthropic API 自动化部署。目前已安装 20 个插件，覆盖从投行 PitchBook 制作到基金月度结账的完整金融业务链条。

📖 项目背景与战略意义

行业背景

金融机构的日常运营依赖大量重复性、高精度的手工工作——分析师花 60% 以上的时间在数据提取、Excel 建模和 PPT 制作上，而非真正的分析判断。传统的自动化工具（RPA、宏脚本）难以应对金融业务中非结构化文档（PDF 报告、电话会记录、邮件）的处理需求。

Anthropic 的战略布局

Anthropic 选择金融服务作为首个行业垂直插件套件，体现了对高价值、强监管行业的战略优先级。该项目解决了三个核心矛盾：

  • 效率与安全的矛盾：金融机构需要 AI 的高效，但必须严格控制数据安全——三层隔离架构（读取层/处理层/输出层）确保不可信文档无法触及写操作
  • 标准化与定制的矛盾：234 个 SKILL.md 技能文件沉淀了机构级的最佳实践（投行 PitchBook 模板、DCF 建模规范、KYC 规则引擎），同时允许每家机构修改 vertical-plugins/ 下的源文件实现差异化
  • 交互与自动化的矛盾：同一份系统提示词既可以交互式使用（Claude Code 插件），也可以自动化部署（Managed Agent API），覆盖"临时分析"和"定时批量处理"两个场景

对金融机构的价值

  • 缩短 Turnaround Time：PitchBook 制作从 2-3 天缩短到分钟级；财报电话会分析从半天缩短到即时
  • 降低操作风险：模型中的每个计算单元格都是公式而不是硬编码（audit-xls 强制检查），对账单逐笔勾稽确保零遗漏
  • 知识资产化：234 个 SKILL.md 将资深分析师的操作经验转化为可复制、可审计的机构知识库

💎 核心价值

🔒 安全优先的三层隔离
每个 Agent 拆分为读取层（仅 Read/Grep，接触不可信文档）→ 处理层（可信隔离区，可连 MCP）→ 输出层（唯一写权限）三层架构，从架构层面杜绝 prompt 注入和数据泄露风险。
🧩 技能与执行分离
Skill（操作手册）和 Agent（执行者）解耦：编辑技能只需改 vertical-plugins/ 下的 SKILL.md，运行 sync-agent-skills.py 自动同步到所有 Agent。一次修改，处处生效。
🔗 多 Agent 接力工作流
通过 handoff_request 协议实现 Agent 间自动接力——Earnings Reviewer 更新模型后自动触发 Model Builder 重建 DCF，形成端到端自动化流水线。
🌐 数据源统一接入
12 个 MCP 数据源（FactSet、CapIQ、LSEG、S&P、Moody's、PitchBook 等）集中配置，所有 Agent 共享。处理层节点在部署时自动获得数据接入能力，无需重复采购。

🎯 典型使用场景

🏦 投行 Pitch 材料制作
MD 提出需求 → pitch-agent 自动从 CapIQ 拉取可比公司数据和先例交易 → 构建 DCF/LBO 模型 → 生成品牌化 PitchBook PPTX → 质量检查 → 交付初稿。原来 2-3 天的工作缩短到 15 分钟。
📰 财报季批量处理
财报季覆盖列表 50+ 公司 → earnings-reviewer 逐个读取电话会记录 → 自动更新覆盖模型中的预测数据 → 撰写研究笔记 → 差异标记。覆盖范围从人工的 5-10 只扩展到全列表，且每份笔记格式统一可审计。
📅 月度结账与对账
gl-reconciler 每日运行：按资产类别比对总账与子账 → 标记差异 → Critic 独立复核 → 输出例外报告。月末 month-end-closer 接收 handoff 自动处理应计项目和滚动前推 → 生成结账包。原本 3-5 天的结账流程压缩到 1 天。
🛡️ 客户开户 KYC 筛查
新客户开户 → kyc-screener 自动解析开户 PDF 中的身份信息 → 运行制裁名单/PEP 规则引擎 → 标记合规缺口 → 输出上报包给合规官。人工审查时间从 2 小时缩短到 15 分钟。
📊 行业研究与创意生成
投资经理提出主题（如"美国数据中心电力供应缺口"）→ market-researcher 自动完成行业概览、竞争格局、可比公司分析 → 输出创意短名单（3-5 家最受益的公司）→ 附带研究报告。从 idea 到 actionable 的短名单不再需要 1 周。
🤝 客户会议准备
客户会议触发 → meeting-prep-agent 自动从 CRM 拉取关系历史和持仓 → 从 CapIQ 获取市场背景 → 读取近期客户通信 → 生成简报包和谈话要点。顾问进会议室前 5 分钟拿到完整资料包。
💡 一句话总结

这不是一套简单的工具集合，而是一个可编程的金融劳动力——每个 Agent 对应一个金融岗位的操作规范，Skill 对应岗位的操作手册，MCP 对应岗位的数据终端，handoff 对应跨岗位的协作流程。

💡 核心发现

10 个智能体覆盖了银行 5 个主要部门（投行部、研究部、运营部、合规部、财务部）的核心业务流程。
所有 Agent 的输出都是草稿或建议，最终决策和分发均在人工控制下——满足金融机构"人机协同"的监管要求。

💡 内网部署核心原则

能封装的不改写：优先将行内现有系统接口封装为 MCP 服务，不修改 Agent 代码
能静态的不实时：非实时场景（如行业研究、PitchBook 模板）用 Excel/CSV 静态注入，减少对接工作量
能分批的不一步到位：先上不依赖外部数据的 Agent（GL Recon、月结），再上有替代方案的 Agent

📌 一句话原则

Agent 是效率工具，不是决策者——所有涉及金额、合规、对外的输出，都必须有人工在环路（Human-in-the-Loop）。

🧩 智能体与技能全景

按项目目录结构分为三大类：智能体（端到端工作流）、垂直领域（技能来源）、合作伙伴（数据连接器）。

💡 技能与 Agent 的关系

所有 Agent 使用的技能都来自垂直插件。 垂直插件按金融领域分类（投行、研究、财务分析、私募等），每个领域包含该岗位所需的专业技能文件（SKILL.md）。

一个 Agent 可以同时从多个垂直插件获取技能。例如 pitch-agent 同时从 investment-banking（投行技能）和 financial-analysis（财务建模技能）加载技能。

垂直插件 = 按领域分类的知识库，Agent = 按工作流跨领域打包的执行体。编辑技能只需改垂直插件下的源文件，运行 sync-agent-skills.py 自动同步到所有引用了该技能的 Agent。

🤖 智能体（Managed Agent Cookbooks）

每个智能体包含完整的 agent.yaml（系统提示词）+ 子代理架构 + 技能集。可直接在 Claude Code 中使用，也可部署为托管代理。

🤖 子代理（Sub-Agent）是什么

每个智能体内部不是一个人干活，而是一个小团队。编排器（编排器/总负责人）接到你的任务后，拆解给多个子代理分工协作。

出于安全考虑，每个子代理的权限被严格限制在三层隔离中：

层级子代理角色权限说明
🥇 读取层 *-reader（资料包/电话会/文档读取器等） Read, Grep 仅 唯一接触外部不可信文档的层级，输出结构化 JSON，绝不写文件
🥈 处理层 *-updater / resolver / builder（模型更新器/核对器等） Read, Grep, Agent, MCP 可信隔离区，不接触原始文档，可连接外部数据源（FactSet/CapIQ）
🥉 输出层 *-writer / poster / publisher（笔记/报告输出器等） Read, Write, Edit 唯一持有写权限的层级，产出最终文件，从不接触不可信文档

returns-analysis — 先收齐进入/融资/运营/退出四类假设，然后构建标准 IRR/MOIC 敏感性表。方法论核心是压力测试：在不同进入倍数、退出倍数和持有期组合下，看 MOIC 和 IRR 的波动范围
portfolio-monitoring — 摄入财务报表后做三层分析：① 提取核心 KPI（收入/EBITDA/杠杆率/现金流）→ ② 对比预算和上期标记差异 → ③ 对超阈值差异追溯根本原因

🧩 核心技能方法论
sector-overview — 金字塔叙事结构：市场规模 ↔ 增长驱动 → 价值链 → 竞争格局 → 变革因素。每个结论都必须有数据支撑，数据必须有来源归属
comps-analysis — 先选 5-8 家最可比的公司，统一指标定义（注意不同会计准则导致的差异），然后用箱线图/百分位法识别异常值，避免被极端值带偏
pitch-deck — 核心原则是"数据绑定"：幻灯片上的每个数字都必须可追溯到 Excel 工作簿中的命名区域。每次修改后做交叉引用验证，确保跨幻灯片的一致性
ib-check-deck — 从四个维度独立审查：① 数字一致性（同一指标在不同幻灯片上是否冲突）→ ② 数据-叙事对齐（图表趋势是否支持文字主张）→ ③ 投行语言标准（术语精确、措辞保守）→ ④ 格式统一性

核心方法论：earnings-analysis 技能

这是 earnings-reviewer 最核心的技能，定义了机构级业绩更新报告（8-12 页，3,000-5,000 字）的分析标准。遵循摩根大通、高盛、摩根士丹利等投行的研究格式。

五阶段工作流程

30-60 分钟
① 数据收集
搜索最新业绩材料，验证发布日期（须在 3 个月内），下载电话会记录、10-Q、新闻稿
2-3 小时
② 分析
每个关键指标的超/低预期分析，细分/地区拆解，利润率与指导分析，更新模型
1-2 小时
③ 图表生成
8-12 个图表：收入/利润率/每股收益趋势、超低预期汇总、估计修订
2-3 小时
④ 报告创建
8-12 页 DOCX，含摘要/结果分析/指标/主题/估值/附录
30 分钟
⑤ 质量检查
验证内容/格式/准确性/及时性，确认每个数字有来源

超/低预期分析三步法

第一步：量化差异
在收入、EBITDA、EPS 三个层面逐一对比实际值 vs 一致预期 vs 先前估计。
格式："收入超预期 1.2 亿美元或 3%"，"EBITDA 低于预期 0.5 亿或 8%"
注意：明确标注一致预期的来源（彭博/FactSet）和日期
第二步：拆解原因
从三个维度解释为什么：
价/量拆分——是涨价还是卖多了？
成本结构——毛利率 vs 运营费用，固定 vs 可变
一次性项目——重组费用、资产减值、税务调整
第三步：更新估计
根据实际结果修订前瞻估计，清晰显示旧估计 vs 新估计。
解释什么变了、为什么变、对投资主题的影响。
是改变主题的事件还是噪音？——区分信号与噪音

引用规范（强制要求 ⭐⭐⭐）

每个数字必须标注特定来源和可点击超链接，来源列表需包含：

✅ 业绩新闻稿（含日期和 URL）
✅ 10-Q 申报文件（含申报日期和 EDGAR 链接）
✅ 电话会议记录（含日期）
✅ 投资者演示/补充材料
✅ 一致预期来源（彭博/FactSet，含日期）
✅ 先前指导（来自上一季度材料）

无来源的数据标记为 [UNSOURCED]，绝不估算。报告末尾附"来源与参考"章节，全部为可点击超链接。

报告结构（12 页模板）

页码内容
第 1 页摘要——评级、目标价、关键要点、超/低预期总览
第 2-3 页详细结果分析——收入/利润拆解、细分地区表现
第 4-5 页关键指标与指导——对比一致预期、管理层指导分析
第 6-7 页更新后的投资主题——业绩对主题的影响、主题是否改变

第 11-12 页附录——图表补充、详细来源列表

模型更新方法论：model-update 技能

这是一个六步执行流程，从识别变化到输出更新后的模型和估值。每个步骤都有标准化的输入/输出格式，确保审计可追溯。

步骤操作输入输出
① 识别变化 确定更新触发因素：业绩发布、指导变化、估计修订、宏观更新、事件驱动（并购/重组/管理层变更） 财报新闻稿、管理层指引 变化类型清单
② 导入新数据 按行项目逐一对比先前估计 vs 实际值，记录差异和备注。需覆盖收入、毛利率、运营费用、EBITDA、每股收益，以及细分/地区的收入和利润率 业绩报告实际值 差异表（行项目/先前估计/实际值/差异/备注）
③ 修订前瞻估计 基于新数据调整未来预测。对比旧 FY 估计 vs 新 FY 估计 vs 旧下一年 vs 新下一年。逐一记录每个假设的变动原因 差异表 + 管理导指引 修订后的财务预测表
④ 重算估值 用更新后的估计重新运行 DCF + 市盈率法 + EV/EBITDA 法。对比先前估值 vs 更新后估值 修订后财务预测 新目标价 + 估值变化表
⑤ 摘要与行动 一段话总结：什么变了、为什么、对股票意味着什么。判断：这是改变主题的事件还是噪音？维持还是更改评级？ 估值变化表 评级/目标价建议 + 执行摘要
⑥ 输出 生成更新后的 Excel 模型 + 估计变化摘要 + 目标价推导说明 全部上述输出 模型文件 + 摘要文档

关键原则

来源先行——在向前预测前，始终将估计与公司报告的实际数字核对，不依赖记忆
区分信号与噪音——单季度波动可能是噪音，趋势变化才是信号。不要在噪音上过度调整
GAAP vs 调整后——注意非经常性项目，明确标注估计是基于 GAAP 还是调整后口径
跟踪修订历史——记录每次估计修订的原因，展示分析演变过程
对照一致预期——修订后的估计与市场共识对比如何？偏差需要合理解释
股份数量陷阱——股票薪酬、可转债、回购带来的稀释可能实质性影响每股收益

🔗 多 Agent 接力工作流

工作原理

接力通过 handoff_request 协议实现，代码在 scripts/orchestrate.py 中：

Agent A
→
{"type":"handoff_request",
"target_agent":"model-builder",
"payload":{"event":"Build dcf..."}}
→
Agent B

步骤：

  1. Agent A（如 Earnings Reviewer）在处理完任务后，在其输出文本中生成一个 handoff_request JSON blob
  2. orchestrate.py 监听 Agent A 的会话输出流（message_delta 事件），用正则 HANDOFF_RE 捕获该 blob
  3. 脚本对 target_agent 做硬编码白名单校验（只允许 10 个已部署 Agent slug）
  4. 对 payload 做 JSON Schema 校验（event 最长 2000 字符、context_ref 正则过滤）
  5. 校验通过后，调用 client.beta.agents.sessions.steer() 向 Agent B 发送引导事件
